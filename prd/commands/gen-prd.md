# PRD(Product Requirements Document)の作成

## Feature File: $ARGUMENTS

徹底的な調査に基づき、機能実装のための完全なPRD(要件定義・設計・実装計画)を作成します。AIエージェントにコンテキストが渡され、自己検証と反復的な改善が行なわれるようにします。

まずはFeature Fileを読み、何を作成する必要があるのか、提供されている例がどのように役立つのか、その他の考慮事項などを理解します。

AIエージェントはあなたがPRDに追加するコンテキストとトレーニングデータのみを取得します。AIエージェントはコードベースにアクセスすることができ、あなたと同じ情報を持ちます。よってあなたの調査結果もしくは参照先がPRDに含まれていることが重要です。AIエージェントはWebにアクセスできるため、ドキュメントやサンプルのURLを渡してください。

## リサーチプロセス

### コードベース分析

- コードベースで類似の機能・パターンを検索する
- PRDで参照するファイルを特定する
- 従うべき既存の慣習・慣例に注意すること

### 外部調査

- 類似の特徴やパターンをオンラインで検索する
- ライブラリのドキュメント
- 実装例(GitHub、StackOverflow、ブログ)
- ベストプラクティスとよくある落とし穴

### ユーザーの説明

- 参考にすべき特定のパターンとその場所はどこか

## PRDの生成

../templates/prd.md をテンプレートとして使用

### AIエージェントに渡すべき重要なコンテキスト

- ドキュメント: 特定のセクションを含むURL
- コードのサンプル: コードベースから得られた実際のスニペット
- 注意点: ライブラリの癖・特徴、バージョンの問題
- パターン: 従うべき既存のアプローチ

### 実装の設計

- アプローチを示す疑似コ_ドから開始
- パターンのための実際のファイルを参照
- エラーハンドリングの戦略を含める
- PRDを満たすために完了すべきタスクを完了すべき順序でリスト化

### 検証方法

- 実装の確認方法およびユニットテストの手段

## アウトプット

リポジトリ内に `.tmp/{YYY-MM-DD}-{feature-name}.md` を作成

### 品質チェックリスト

- [ ] 必要なコンテキストがすべて含まれている
- [ ] 検証方法がAIによって実行可能
- [ ] 明確な実装パス
- [ ] エラーハンドリングの文書化

PRDを1から10のスケールで評価

重要: 目標は包括的なコンテキストを通じてのone pass実装の成功です
